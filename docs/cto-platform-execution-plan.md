# CTO 技术中台实施计划

Status: Active  
Last updated: 2026-03-13

## 1. 目标

这份计划对应的是“从当前 Stage B 中后段继续推进到最终理想状态”的执行蓝图。

目标不是继续围绕小说场景打补丁，而是把当前能力收成一个真正可运营的通用平台：

- `Resource` 成为唯一业务真相源
- `App` 成为公司内唯一正式入口
- `AppManifest` 成为所有正式 App 的数据与动作契约
- `SkillDefinition` 作为内部执行模型，支撑对外“能力/工具”
- `CapabilityRequest / CapabilityIssue` 成为治理回路

一句话：

**让 CTO 团队提供的能力，能够被发布、发现、触发、反馈、修复、验证，并在多个业务场景下复用。**

## 2. 当前状态判断

截至 2026-03-13，当前系统已经明显超出 MVP：

- 显式 App、通用 AppManifest、Resource tags、能力定义、需求/问题治理、运行台账、嵌入式宿主壳都已经落地
- 小说场景已经走通
- 非小说 `内容工厂` 基线场景已经走通
- 公司创建已经支持 degraded provisioning，不再因 OpenClaw executor 暂时不可见而整单失败

当前不再属于“没有模型”的问题，而是 4 个更后期的问题：

1. Resource 的业务主路径还没有完全只依赖显式声明
2. 能力执行还残留少量平台内置语义桥接
3. 跨场景通用性验收还不够广
4. 最终版运维与审计清单还未完全关单

## 3. 总体推进顺序

按下面顺序推进，不建议并行拉太多面，否则会继续在局部看起来很强、整体却难关单。

### Phase 1. 显式资源收口

目标：
- 让 `resourceType / tags / manifest` 成为唯一业务真相源

完成标准：
- 预检、规则判断、阶段建议、发布检查都只消费 `declared/manifest` 资源
- `inferred` 资源只用于展示、草案、空状态补位
- UI 对推断资源有清晰提示，并给出“补 manifest / 发布为正式资源”的入口

关键任务：
- 盘点所有仍在读取推断资源参与业务逻辑的路径
- 按路径逐个替换为显式资源过滤
- 为资源视图模型补齐来源标识和说明文案
- 补测试，确保 inferred 资源不能满足正式判断

### Phase 2. 能力执行引擎收口

目标：
- 让 `entryPath + input contract + output contract + adapter` 成为统一执行主路径

完成标准：
- 固定优先级为：
  - `workspace script`
  - `registered adapter`
  - `honest failure`
- 没有脚本和 adapter 的 `ready` 能力必须失败并记录问题，不再伪成功
- `manual smoke test` 与正式触发共享同一执行协议

关键任务：
- 继续把内置能力桥接迁移到正式 adapter registry
- 审查 runner 中残留的硬编码语义
- 完善输入输出契约文档和执行摘要
- 补执行失败、回退、阻塞与 issue 生成测试

### Phase 3. 第二异构场景验收

目标：
- 证明平台不只是“小说 + 内容工厂”双场景勉强兼容

候选场景：
- 游戏模拟器
- 非内容型运营控制台
- 数据检查/回放工具

完成标准：
- 不新增平台级主对象
- 走通以下完整链路：
  - 显式 App 注册
  - AppManifest 驱动展示
  - 显式 Resource 供给
  - 能力执行并写回结果资源
  - 请求/问题治理
  - 待验证闭环

验收重点：
- 模板体验与底层平台对象分层是否成立
- 非小说语义下是否仍有小说专属词残留
- 能力执行是否仍依赖小说默认逻辑

### Phase 4. GA Closeout

目标：
- 对照最终标准逐条关单，完成 Stage C / Final / GA

完成标准：
- 满足 [docs/cto-platform-delivery-standard.md](/Users/zkk/openclaw/cyber-company/docs/cto-platform-delivery-standard.md) 第 3 节定义的 10 条标准
- 至少 3 类异构业务场景通过同一套平台验收
- 平台运维、审计、台账、失败诊断、恢复说明齐备

关键任务：
- 完成最终验收矩阵
- 补运维与审计视图缺口
- 清理仍然只是临时兼容的旧分支和兜底
- 完成 Stage C 关单文档

## 4. 本轮推荐执行项

按当前状态，接下来最值的 4 件事是：

1. 盘点并收掉所有“推断参与业务判断”的路径
2. 继续削弱 built-in adapter 的语义桥接
3. 选择并接入第二个异构场景
4. 预先补运维/审计 closeout 清单，避免最后集中返工

## 5. 风险与边界

### 5.1 不能再让推断进入业务真相源
- 猜测适合做草案，不适合做判断
- 任何正式结论都必须建立在显式资源和 manifest 上

### 5.2 不能把“能力执行”重新做回产品特例
- 内置能力可以保留 adapter
- 但语义必须逐步回到 contract + registry，而不是越来越多 if/else

### 5.3 不能只在小说场景里优化体验
- 小说体验可以继续保留
- 但接下来每一轮都要问：
  - 非小说场景是否也成立
  - 如果换游戏/运营/数据工具，是否仍无需新增平台级主对象

## 6. 当前完成定义

这一阶段完成的判断不是“又做了几个入口”，而是：

- 平台对象没有继续发散
- 显式资源彻底替代推断成为业务真相源
- 能力执行真正遵守统一 contract
- 至少再多一个异构场景证明平台通用
- 最终版 closeout 不再需要大规模返工
