# 产品下一步：让 Cyber Company 真正好用

Status: Draft
Last updated: 2026-03-14

## 文档目的

这份文档回答一个问题：**从产品视角，现在最应该做什么才能让系统真正好用？**

不是代码重构。不是架构优化。是**用户打开系统后能不能顺利用起来**。

基于以下研究输出：
- 当前代码库的产品分析
- 用户体验审查 (`current-product-user-experience-review.md`)
- NL 小说公司场景推演 (`nl-novel-company-guidance.md`)
- Paperclip 对比分析

---

## 当前产品的核心问题

用一张图说清楚：

```
你想让用户这样：   CEO 首页 → 提需求 → 看进度 → 验收结果 → 开心走人
                     ↓         ↓       ↓         ↓
用户实际体验：     7个入口不知道先点哪个 → 看到一堆运行态/Ops → 
                  验收按钮不知道为什么能按 → Agent 卡了不知道怎么办
```

**问题不是功能不够多，是已有功能没有串成一条让用户走得通的路。**

---

按"用户感知改善 × 实现复杂度"排序，**不是按代码优雅度排序**：

| 优先级 | 改什么 | 用户感知 | 复杂度 | 预计工作量 |
|--------|--------|---------|--------|-----------|
| **P0** | 砍导航，收口主路径 | ⭐⭐⭐⭐⭐ | 🔧 低 | 1-2 天 |
| **P0** | CEO 自主巡检（heartbeat） | ⭐⭐⭐⭐⭐ | 🔧🔧🔧 中高 | 3-5 天 |
| **P1** | 验收证据可视化 | ⭐⭐⭐⭐ | 🔧🔧 中 | 2-3 天 |
| **P1** | 一键接管替代文本包 | ⭐⭐⭐⭐ | 🔧🔧 中 | 2-3 天 |
| **P2** | Workspace 默认改为交付视图 | ⭐⭐⭐ | 🔧 低 | 1-2 天 |

| **P3** | 术语统一 | ⭐⭐ | 🔧 低 | 1 天 |

## P0-1：砍导航，收口主路径

### 现状

当前主导航暴露了 7+ 个一级入口：CEO 首页、需求中心、看板、运营大厅、运行态、工作目录、设置。

用户打开系统的第一反应是"去哪？"而不是"开始吧"。

### 目标

普通用户只看到 3 个一级入口：

```
┌─────────────────────────────────────────┐
│  CEO 首页  │  需求中心  │  设置        │
│  (发起)    │  (推进+验收) │  (偶尔调参)  │
└─────────────────────────────────────────┘

其他入口降级为：
  - 运行态 → 异常时在需求中心内 contextual 引导
  - 运营大厅 → 需求中心补充面板或高级入口
  - 看板 → 需求中心的执行摘要
  - 工作目录 → 需求中心"看交付"时跳转
```

### 具体改动

1. **左侧导航**：默认态只显示 CEO 首页、需求中心、设置
2. **高级入口**：看板/Ops/运行态/工作目录收到"更多"或"高级"折叠区
3. **CEO 首页**：移除"打开运营大厅""查看运行态"等首屏按钮
4. **需求中心**：当系统检测到异常时，才在页内出现"去排障"的 contextual CTA

### 不做什么

- 不删除这些页面——只是不默认暴露
- 不改变页面内部逻辑——只改导航层级和入口位置

### 验收标准

- 新用户打开系统，3 秒内能判断"我该在哪里开始"
- 没有异常时，用户不会看到运行态/Ops 相关入口

## P0-2：CEO 自主巡检（Heartbeat）

### 现状

CEO 在收到用户"可以开始"后能派单。但回执返回后，CEO 不会主动触发下一轮——需要用户手动"推"。

这直接破坏了"AI 自主驱动"的核心承诺。

### 目标

CEO 像一个真正的总经理——**不需要老板每次戳他，自己就会检查进度并推下一步**。

### 设计方案

```
CompanyOpsEngine 增加 CEO 巡检轮次：

每 N 分钟（配置化，建议默认 5 分钟）：
  1. 扫描所有 status=answered 的 dispatch
     → 如有：CEO 自动读取回执，决定下一步
  
  2. 扫描所有超时未回执的 dispatch
     → 如有：触发 SLA 升级（已有逻辑）
  
  3. 扫描需求中心的当前主线状态
     → 如果所有子任务完成：CEO 自动发起阶段性汇报
     → 如果有阻塞：CEO 自动评估是否需要换人或请求支持
  
  4. 写入巡检日志
     → 用户可在需求中心看到"CEO 上次巡检时间"和"下一步计划"
```

### 关键约束

- 巡检不等于无限循环——每轮有明确的"检查→决策→执行→等待"
- 有预算护栏——单次巡检产生的 token 消耗受 `automationMonthlyBudgetUsd` 约束
- 有紧急刹车——用户在设置中可以暂停自动巡检
- 以 Cyber Company 自身系统为 heartbeat 的单一权威源；如沿用 OpenClaw heartbeat，只允许由系统单向同步配置，不再维护第二套业务 heartbeat 设置

### 借鉴 Paperclip

Paperclip 的 heartbeat 模式值得参考：
- `heartbeat_runs` 表记录每次心跳执行（时间、状态、结果）
- `heartbeat_run_events` 记录心跳内发生的具体事件
- 有 `succeeded / failed / cancelled / timed_out` 终态

建议用类似思路给 CEO 巡检加审计：

```typescript
interface CeoHeartbeatRecord {
  id: string;
  timestamp: Date;
  status: 'succeeded' | 'failed' | 'timed_out';
  dispatches_checked: number;
  actions_taken: string[];   // ["read_report_from_writer", "dispatched_to_editor"]
  next_check_at: Date;
  token_cost: number;
}
```

### 验收标准

- 用户提完需求并确认后，即使离开系统 1 小时，回来能看到 CEO 已经在推进
- 需求中心能看到"CEO 上次巡检 5 分钟前，已完成 3 次派单"
- 巡检有可追溯的日志

## P1-1：验收证据可视化

### 现状

需求中心的"验收通过"按钮边上没有任何证据告诉用户：
- 做了什么
- 检查过了什么
- 为什么可以按

用户按下去完全靠猜。

### 目标

验收按钮旁边有一个**验收依据面板**：

```
┌──────────────────────────────────────┐
│  验收依据                            │
│                                      │
│  📦 交付物                           │
│  ├── chapter-01.md  (已生成)         │
│  ├── chapter-02.md  (已生成)         │
│  └── worldview.md   (已更新)         │
│                                      │
│  ✅ 检查通过                         │
│  ├── 校对编辑已审核 (通过)           │
│  └── 设定一致性检查 (通过)           │
│                                      │
│  ⚠️ 注意项                           │
│  └── 第2章节奏评分偏低 (72/100)      │
│                                      │
│  📊 本轮统计                         │
│  ├── 派单 12 次                      │
│  ├── 返稿 3 次                       │
│  └── 耗时 45 分钟                    │
│                                      │
│  [验收通过]  [继续修改]  [驳回重开]   │
└──────────────────────────────────────┘
```

### 具体改动

1. **数据来源**：从已有的 Workspace artifact、dispatch history、closeout 检查聚合
2. **不需要新模型**：先用现有的 workspace 文件列表 + dispatch 统计 + closeout 检查结果拼
3. **展示位置**：需求中心验收区，紧贴验收按钮上方

### 验收标准

- 用户在按"验收通过"前，能看到至少 3 条具体的交付证据
- "为什么可以通过"有可见、可理解的答案

## P1-2：一键接管替代文本包

### 现状

Agent 卡住后，用户需要：打开会话 → 找到接管包 → 复制文本 → 手动继续。

### 目标

Agent 卡住后，用户看到的是：

```
┌──────────────────────────────────────┐
│  ⚠️ 写手（阿墨）在"第5章初稿"上阻塞  │
│                                      │
│  原因：连续 2 次返稿修改后陷入循环    │
│  已卡住：15 分钟                      │
│                                      │
│  [重试]  [换人处理]  [我来接管]       │
└──────────────────────────────────────┘
```

三个按钮：
- **重试**：清空当前 dispatch 状态，重新派给同一个 agent
- **换人处理**：系统自动选择候选 agent（如另一个写手），重新派单
- **我来接管**：用户直接进入该会话，带有当前上下文摘要

### 具体改动

1. **触发**：OpsEngine 已有的 escalation 检测
2. **展示**：需求中心或 CEO 首页的"待处理"区域
3. **动作**：复用现有的 re-dispatch 和 takeover pack 逻辑，只是 UI 化

### 验收标准

- Agent 卡住时，用户一键就能让系统恢复运转
- 不需要理解 dispatch、takeover pack 等内部概念

## P2-1：Workspace 默认改为交付视图

### 现状

用户进 Workspace 首屏看到的是 AppManifest、Capability Binding、Workflow 等平台治理概念。

### 目标

默认首屏只显示交付物：

```
Tab 1（默认）: 交付物
  - 已生成的章节文件列表
  - 最新版本 diff
  - 校对标注

Tab 2: 设定管理
  - 世界观文档
  - 角色关系图
  - 时间线

Tab 3（高级，折叠）: 平台管理
  - AppManifest / Capability / Workflow binding
```

### 验收标准

- 非技术用户进 Workspace，3 秒内能看到"我的小说写到哪了"

---

## P2-2：Ops 语义收口

### 现状

运营大厅同时承载排障、审批、团队管理、快速派活、知识同步、招聘/解雇。

### 目标

拆成两个概念：
- **异常中心**（从需求中心引导进入）：阻塞、超时、接管、恢复
- **组织控制**（从设置页访问）：审批、成员调整、知识同步、Blueprint

### 验收标准

- 用户被引导进 Ops 时，知道自己是来"处理异常"而不是"管理公司"

## P3：术语统一

把以下混用收口：

| 当前混用 | 统一为 |
|---------|--------|
| Operations Hall / 运营大厅 / Ops | **异常中心** |
| 工作目录 / Workspace / 公司应用 | **工作目录** |
| 主线需求 / 当前需求 / 目标 | **当前主线** |

改动范围：README、导航文案、空状态文案、卡片标题、glossary。

## 实施建议

### 第一周：最高性价比

```
Day 1-2: 砍导航（P0-1）
  → 改 App.tsx 和导航组件，隐藏高级入口
  → 改 CEO 首页，移除多余 CTA
  → 效果：用户打开系统立刻清爽

Day 3-5: CEO Heartbeat（P0-2）
  → 在 CompanyOpsEngine 增加 CEO 巡检轮次
  → 加巡检日志展示
  → 效果：系统开始自己转了
```

### 第二周：信任感建立

```
Day 6-8: 验收可视化（P1-1）+ 一键接管（P1-2）
  → 需求中心增加验收依据面板
  → 异常时显示一键操作按钮
  → 效果：用户敢按验收按钮了，Agent 卡了不慌了
```

### 第三周：收尾打磨

```
Day 9-10: Workspace 默认改交付视图（P2-1）+ Ops 收口（P2-2）
Day 11: 术语统一（P3）
```
