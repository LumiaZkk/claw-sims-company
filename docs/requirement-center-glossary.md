# Requirement Center Glossary

Status: Draft  
Last updated: 2026-03-10

## 主线需求

当前系统唯一正在推进的 primary requirement。  
前台所有核心页面必须围绕同一条主线显示状态。

## Requirement Aggregate

本地主线权威对象。  
字段至少包含：`topicKey / owner / stage / nextAction / roomId / status / acceptanceStatus`。

## 执行完成

团队已经把当前任务结果推进到可收口状态。  
这不等于用户已经认可结果。

## 待你验收

用户必须做出决定的状态。  
此时不应再默认继续派单，而应由用户选择通过、继续修改或驳回重开。

## 验收通过

用户明确确认结果满足目标，需求正式闭环。

## 驳回重开

用户认为当前结果不达标，主线重新回到执行态。  
这是一条已知主线的重开，不应创建第二条 primary requirement。

## 继续修改

用户认可方向但认为结果还不够，需要继续优化后再次提交验收。  
前台语义上回到执行态，不应停留在已完成。

## 需求房

当前主线的协作视图。  
它可能映射到多个成员 direct session 的投影，不等于所有 provider 上天然存在的一条原生群聊线程。

## 派单透明化

在需求房中明确展示：

- 派给谁
- 谁回复了
- 当前消息对应哪次 dispatch

## 交付物

当前主线相关的文件、报告、设定和工具产出。  
完整视图由 Workspace 承载，需求中心只显示摘要。

## Ops

异常与排障中心。  
负责超时、失联、请求闭环、交接阻塞等恢复工作，不负责主线叙事。
